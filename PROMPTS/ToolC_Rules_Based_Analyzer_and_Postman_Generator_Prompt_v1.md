# ToolC_Rules_Based_Analyzer_and_Postman_Generator_Prompt_v1.md

You are a senior Python engineer building a rules-based API endpoint classifier and Postman collection generator.

---

## 工具鏈定位

```
PHP 專案
  → [ToolA v2]  → features_raw.jsonl + features_report.md
  → [人工審閱]   → 確認信號、排除誤報
  → [Claude Code + ToolC prompt] → 生成 pattern.json 草稿
  → [人工確認]   → 最終 pattern.json
  → [ToolC]     → postman_collection.json + endpoint_catalog.json
```

ToolC 是工具鏈的**最終執行階段**，不做任何 PHP 原始碼掃描，只做分類與生成。

---

## Goal

Build a Python CLI tool ("tool_c") that reads:
1. A `features_raw.jsonl` file produced by **ToolA v2** (`schema_version: "2.0"`)
2. A user-confirmed `pattern.json` rules file (generated by Claude Code and human-reviewed)

And then:
- Re-classifies PHP files as API endpoints using `pattern.json` rules applied to JSONL evidence
- Infers HTTP method per endpoint using multi-source priority logic
- Matches response envelope templates from `pattern.json`
- Generates a Postman Collection v2.1 JSON file ready for direct import
- Optionally outputs a structured `endpoint_catalog.json` for audit

---

## Key Design Principle

ToolC does **not** re-scan PHP source files. It is a **pure classification and generation engine** that operates entirely on `features_raw.jsonl`:

- All PHP file I/O and regex detection is done by ToolA
- ToolC reads pre-computed structured evidence from JSONL
- The PHP project root is **not required** as input
- Re-scoring is done by applying `pattern.json` weights to signals already extracted by ToolA

---

## Upstream JSONL Schema Contract

ToolC consumes `features_raw.jsonl` produced by ToolA v2 (`schema_version: "2.0"`). The file has three record types in **fixed order**:

| Position | `record_type` | Purpose |
|---|---|---|
| Line 0 | `global_stats` | Framework info, signal frequency table, helper registry, generation hints |
| Lines 1–N | `file` | Per-file signals, route hints, params, envelope keys, ToolA score |
| Last line | `skipped_files_summary` | Files ToolA skipped (too large / encoding errors) |

ToolC startup sequence:
1. Read Line 0 (`global_stats`). Validate `schema_version == "2.0"`. If mismatched → exit with code 1 and clear error.
2. Extract `global_stats.framework.detected` for framework-aware logic.
3. Use `global_stats.signal_frequency_table` to cross-validate signals declared in `pattern.json`.
4. Read `global_stats.pattern_json_generation_hints` and compare against `pattern.json` thresholds — warn if divergence exceeds 15 points (warn only, do not override).
5. Stream Lines 1–N for classification.
6. Read last line (`skipped_files_summary`) and carry forward to `endpoint_catalog.json`.

### Schema Validation Rules (Enforced at Startup)

All validations run before any classification begins. Failures abort with exit code 1. Warnings are printed but execution continues.

| # | Rule | On Fail |
|---|---|---|
| V1 | All `pattern` values must compile via `re.compile()` | Exit 1 — report field path + invalid regex |
| V2 | `thresholds.uncertain` < `thresholds.endpoint` strictly | Exit 1 |
| V3 | `weight` in `negative_signals` must be negative integer | Exit 1 |
| V4 | `weight` in `strong_signals` and `weak_signals` must be positive integer | Exit 1 |
| V5 | Signal `name` must exist in `global_stats.signal_frequency_table[].signal` or `custom_helper_registry[].helper_name` in JSONL | Warning: `"WARNING: signal '{name}' not observed in JSONL — may produce 0 matches"` |
| V6 | `envelope.example` object keys must be a subset of `keys_all_of` + `keys_any_of` | Exit 1 |
| V7 | `framework` in pattern.json must match `global_stats.framework.detected` in JSONL | Warning: `"WARNING: framework mismatch — pattern.json='{x}', JSONL='{y}' — proceeding with pattern.json value"` |
| V8 | `endpoint_envelopes.templates[].name` values must be unique | Exit 1 |
| V9 | `method_inference.priority_order` must contain `"default"` as last element | Exit 1 |
| V10 | `thresholds.endpoint` diverges from JSONL `pattern_json_generation_hints.recommended_endpoint_threshold` by more than 15 | Warning: `"WARNING: threshold divergence — your value={x}, JSONL hint={y}"` |
| V11 | Any top-level key in `pattern.json` that begins with `_` (underscore) is treated as an **extension field** and silently ignored. ToolC must **not** raise a validation error for these keys. This allows ToolB's `_tool_b_meta` block to pass through without modification. | Ignore (no error, no warning) |
| V12 | `(thresholds.endpoint - thresholds.uncertain)` must be ≥ `minimum_threshold_gap` from JSONL `pattern_json_generation_hints` (default: 10 if field absent). | Warning: `"WARNING: threshold gap {gap} is below recommended minimum {min_gap} — classification boundary may be too narrow"` |
| V13 | `pattern.json` field `source_jsonl_schema_version` must equal `"2.0"` (the schema version of the JSONL this pattern was derived from). If missing or mismatched → Warning: `"WARNING: pattern.json source_jsonl_schema_version='{x}' does not match expected '2.0' — pattern.json may have been generated from a different ToolA version"`. Do not Exit 1: the JSONL itself is already validated at startup (step 1). This is a self-declared version consistency check only. |

## Classification Engine

ToolC re-scores each `file` record from the JSONL using `pattern.json` rules. **ToolA's score is used only as a cross-check reference**, not as the primary score.

### Re-scoring Algorithm

```python
def toolc_score(file_record, pattern):
    score = 0

    # Strong signals — cap per-signal at weight * 3 to prevent inflation
    for sig in file_record["signals"]["strong"]:
        match = find_signal_in_pattern(sig["name"], pattern["scoring"]["strong_signals"])
        if match:
            contribution = match["weight"] * min(sig["occurrences"], 3)
            score += contribution

    # Weak signals — no multiplication (occurrences not amplified)
    for sig in file_record["signals"]["weak"]:
        match = find_signal_in_pattern(sig["name"], pattern["scoring"]["weak_signals"])
        if match:
            score += match["weight"]

    # Negative signals
    for sig in file_record["signals"]["negative"]:
        match = find_signal_in_pattern(sig["name"], pattern["scoring"]["negative_signals"])
        if match:
            score += match["weight"]  # negative value

    # Custom helpers called (treated as strong signals if registered)
    for helper in file_record["custom_helpers_called"]:
        match = find_signal_in_pattern(helper["name"], pattern["scoring"]["strong_signals"])
        if match:
            score += match["weight"]

    return max(0, min(100, score))  # clamp to [0, 100]
```

### Cross-Check with ToolA Score

After re-scoring, compare `toolc_score` with `file.score` (ToolA's heuristic score):

- If `abs(toolc_score - toola_score) > 30`: record `score_divergence_warning` in catalog
- **Do not override ToolC's score.** The warning is for human review only.
- Log divergence summary in dry-run output.

### Confidence Tiers

| ToolC Score | Tier | Label | Included in Postman by default |
|---|---|---|---|
| ≥ `thresholds.endpoint` | **L1** | `confirmed_endpoint` | ✅ Yes |
| ≥ `thresholds.uncertain` and < `thresholds.endpoint` | **L2** | `uncertain` | Only with `--include-uncertain` |
| < `thresholds.uncertain` | **L3** | `not_endpoint` | ❌ Never |

## HTTP Method Inference — Multi-Source Priority

Method inference follows `method_inference.priority_order` from pattern.json. Each source is attempted in order; the first successful determination wins.

### Source 1: `route_hints` (highest priority)

If `file.route_hints` is non-empty and contains entries with `confidence: "high"`:
- Use the `method` from each route hint directly.
- If a file has **multiple route hints** (e.g., both GET and POST), generate **one Postman item per route hint**.
- Item name format: `"[METHOD] [uri] ([controller_method if available])"` — e.g., `"POST /api/users (UserController@store)"`.

If `confidence` is `"low"` or `"medium"`, treat as Source 4 (signal_based) fallback only.

### Source 2: `request_method_check`

If `file.method_hints` contains entries where `evidence` matches `$_SERVER['REQUEST_METHOD']` checks, use the extracted method.

### Source 3: `input_param_type`

Apply these rules in order:
- Has `json_body` params and no `get` params → `POST`
- Has only `get` params and no `json_body` params → `GET`
- Has both `json_body` and `get` params → generate **two Postman items**: one GET (with query params), one POST (with JSON body)

### Source 4: `signal_based`

Apply `method_inference.rules` from pattern.json. Match on `matched_signal_name` against `file.signals.strong[].name`.

Example rule:
```json
{
  "source": "signal_based",
  "condition": "WordPress AJAX handler — always POST",
  "matched_signal_name": "add_action('wp_ajax_",
  "method": "POST"
}
```

### Source 5: `default` (lowest priority)

Use `method_inference.default_method` from pattern.json. Always record inference source as `"default"` in catalog and Postman metadata.

**Multi-item files**: When a file generates more than one Postman item (multiple route hints or conflicting param types), each item must have a unique name. If names would collide, append a disambiguator: `"GET /api/users [1]"`, `"GET /api/users [2]"`.

## Envelope Template Matching

For each L1 and L2 file, attempt to match `file.envelope_keys[].key` against `endpoint_envelopes.templates`:

```python
def match_envelope(file_envelope_keys, templates):
    key_set = {e["key"] for e in file_envelope_keys}
    for template in templates:
        all_present = all(k in key_set for k in template.get("keys_all_of", []))
        any_present = any(k in key_set for k in template.get("keys_any_of", []))
        if all_present and any_present:
            return template
    return None
```

If matched: include `template.example` as Postman example response body.
If no match: use `{}` as example body, add note in `_tool_c_meta`: `"no_envelope_match: true"`.

#### Body Generation Rules

| Method | Has `json_body` params | Has `post` params | Body mode |
|---|---|---|---|
| POST / PUT / PATCH | ✅ | any | `raw` JSON skeleton from `json_body` keys |
| POST / PUT / PATCH | ❌ | ✅ | `urlencoded` from `post` keys |
| POST / PUT / PATCH | ❌ | ❌ | `raw` with `{}` and note `"no params extracted"` |
| GET / DELETE | any | any | No body |

JSON skeleton values use typed placeholders based on key name heuristics:
- Keys ending in `_id`, `id`, `count`, `page`, `limit` → `0`
- Keys ending in `_at`, `date`, `time` → `""`
- Keys matching `is_`, `has_`, `enabled`, `active` → `true`
- All other keys → `""`

#### Query Params Generation

For GET items, populate `url.query` from `file.input_params.get`:
```json
{ "key": "id", "value": "", "disabled": false }
```

#### Secret Redaction in Postman Items

Before writing any `body.raw` or `response[].body`, apply the same redaction rules as ToolA:
- Matches for `(api_?key|token|secret|password)` patterns → replace value with `"REDACTED"`
- Set `_tool_c_meta.redaction_applied: true` if any redaction occurred

#### Folder Structure (`--folder-structure by_directory`)

Group items into Postman folder objects by their top-level directory path:
```json

  "item": [ "... items from this directory ..." ]
}
```

Files at root level go into a `"(root)"` folder.

## Implementation Notes

- **Stream JSONL**: Read line by line. Do not load entire JSONL into memory. Buffer only `global_stats` (Line 0) in memory for cross-referencing.
- **Stable sort**: All output items in `postman_collection.json` and `endpoint_catalog.json` must be sorted by `file` path (ascending), then by `method` (alphabetical), then by `uri` (alphabetical). Same input always produces identical output.
- **Postman schema validation**: After building the collection, validate it against the official Postman Collection v2.1 JSON Schema (`https://schema.getpostman.com/json/collection/v2.1.0/`). Cache the schema locally; do not require network access at runtime (embed schema in package).
- **No hardcoded URLs**: Never embed literal hostnames or IP addresses in the output. Always use `{{baseUrl}}`.
- **Error exit codes**: `0` = success, `1` = validation error, `2` = JSONL schema mismatch, `3` = I/O error.
- **Encoding**: Read JSONL with `encoding='utf-8'`. If a line fails to parse as JSON, log the line number and skip — do not crash.
- **Extension fields (`_` prefix)**: When loading `pattern.json`, strip all top-level keys beginning with `_` before passing the object to jsonschema validation. These are reserved for toolchain metadata (e.g. ToolB's `_tool_b_meta`). Stripped keys must be logged at DEBUG level: `"Ignored extension field: _tool_b_meta"`. Never raise an error for extension fields (V11).
- **Shared validator module**: The validation logic for `pattern.json` (V1–V13) must be implemented in a standalone module named `toolchain_validator.py`, placed in a shared location importable by both ToolB and ToolC. Neither tool should contain its own copy of these rules. The module exposes a single entry point: `validate_pattern_json(pattern: dict, global_stats: dict) -> ValidationResult`. Any future change to V1–V13 is made only in `toolchain_validator.py` and takes effect in both tools immediately.
- **`minimum_threshold_gap` fallback**: If `pattern_json_generation_hints.minimum_threshold_gap` is absent from JSONL (e.g. older ToolA output), default to `10` for V12 check.
